横版平台跳跃游戏跳跃动作的分析与复现

项目目的：Unity自带的重力系统表现平台跳跃的跳跃动作有一定局限性，由此分析部分平台跳跃游戏的移动和跳跃形式，希望实现体验更加自然的跳跃动作，以此为加入校内游戏工作室的平台跳跃射击游戏Demo进行准备。

项目工作：

  1. 搜集资料或自行录制，对经典平台跳跃游戏的移动、跳跃的动作特征进行定性的描述
  2. 对这些动作特征进行分解，考虑程序实现的可能性
  3. 在Unity中实现分解动作及其配合

项目结果：为入部Demo实现了一个相对自然的跳跃系统，包括：

  • 可通过按键时间控制跳跃高度
  • 脉冲+加速实现的上升段，有速度限制的下落段
  • 部分遵循但不完全模拟力学原理，体验上更加自然的加减速过程
  • 跳跃贴边时顺着墙边继续上升至预订高度

横版平台跳跃的动作系统大多集中在水平和垂直两个维度上，就水平维度而言相对简单，主要是控制角色的移动方向和移动速度，一些手感较好的游戏会有明显的加减速过程、惯性和地面摩擦感。

基本水平移动

在Unity中，使用Input.GetAxis("Horizontal")能够获取输入设备传入的水平方向向量，其值被标准化到$[-1,1]$区间，且输入发生变化时能在突变的输入量间进行平滑插值，模拟一种有“缓冲感”的输入。

例如，在上述代码块中，如果玩家在操作中忽然从按住D键变向按住A键，那么h将会在短时间内从1平滑变化到-1，直观体验上，玩家便不会直接调头并且获得反向的速度。如果使用的是Input.GetAxisRaw，获得的h_raw则只可能是-1、0、1三个取值，玩家按反方向键就会发生直接调头并获得全部速度。以上，使用GetAxis是一种能够实现有简单惯性、简单平滑速度效果的解决方案。

这是一条典型的抛物线轨迹，是以时间 $t$ 为唯一变量的函数，而其约束条件是一次项中的初速度 $v_0$ 。如果在游戏中单纯套用这个公式的话，表示玩家起跳的那一刻，跳跃轨迹的垂直分量就已经注定了，但这样真的可行吗？提出如下问题：

  • 玩家跳跃欠缺空间自由度：如果初速度 $v_0$ 重力加速度 $g$ 固定，那么玩家无论如何都会在固定时间达到最大高度，面对不同的平台高度，玩家并没有控制跳跃高度——即空间层面的“自由度”。
  • 玩家跳跃欠缺时间自由度：同理，如果玩家面临前方有高低不一的障碍、敌人等情况，过短的腾空时间会跳不过去，过长的腾空时间则又有可能撞到后面的障碍，那么玩家必然会确实控制自己跳跃时间的——时间层面的“自由度”。
  • 下落速度会无限加速：即使考虑到现实中的下落，重力也最终会和风阻力达到受力平衡而不再加速。那么游戏中如果玩家下落速度无限增大，尤其是在地图本身有极大纵向高低落差的情况下，持续加速下落会让玩家越来越难以观察下方的情况。

案例分析

针对以上问题，我结合了一些网络上的录像资源和自己的游戏资源，对一部分较热门的平台跳跃游戏的一段跳动作进行了进行分析。

空洞骑士

空洞骑士的跳跃主要被用于通过地图、机关规避、战斗躲避等，使用频率相当高。玩家跳跃时可以在横轴上进行无惯性瞬间转向，跳跃在水平方向的可操控性也是与游戏要求的频繁位移相对应的。而在纵向上，空洞骑士的跳跃高度相比角色身位来说相当高，跳跃曲线更像一个开口较小的抛物线，或者说是“ $\and$ ”形的，如图1，角色的滞空时间并不会太长，但玩家能利用一段跳滞空其间衔接其他动作，包括与二段跳、劈砍、技能释放等。也即空洞骑士的跳跃主要在地图行动、紧急位移两方面发挥作用，因此为了玩家能够顺利过图、有效与敌人战斗，它的跳跃机制必须给玩家提供较大的控制空间。

图1 空洞骑士 跳跃

上升段：

  • 可以通过按键时间控制跳跃高度
  • 停止按键或持续按键一段时间后，有一段比较短时间的减速上升
  • 短按跳跃时减速段明显更短，考虑为初速度不同

滞空期：

  • 可进行二段跳、攻击、技能的衔接，在跳跃抛物线最高点释放可控性最高

下降段：

  • 有明显的下落加速，和起跳轨迹比较对称

奥日系列

奥日的跳跃系统相当流畅，在我自己的游戏经历中个人认为属于顶尖水平。与空洞骑士不同，奥日跳跃的滞空时间相当长，对应的也是奥日在地图上丰富的纵向层次、机关和极多的动作衔接，所以，奥日的跳跃曲线是一个开口更大的抛物线，有点像“ $\cap$ ”形状。此外，奥日的跳跃有一种“灵动”感，即能让玩家体验到跳跃过程中的速度感，但同时这种速度感主要在视觉上进行呈现，而非直接让玩家操作一个速度极快以至于难以控制的物体，也即让玩家能从视觉层面获得超出自身操作预想的实际体验。

图2 奥日与萤火意志 长距离下落

图3 奥日与萤火意志 连续跳跃

上升段：

  • 可以通过按键时间控制跳跃高度，且不同跳跃高度、是否连跳会带来不同角色动画的呈现
  • 起跳有极短时间的起跳过程，此时角色没有马上离地，但视觉上可看出再发力蹬地
  • 离地瞬间产生明显的脉冲速度，之后迅速进入减速状态，开始纵向速度较低的滞空期

滞空期：

  • 滞空期较长，可衔接二段跳，触发攻击技能的空中姿态，衔接部分动作系统的机制对策

下降段：

  • 到达最大高度后以零初速度开始向下加速，加速过程非常明显
  • 如果长距离下落，下落速度并不会一直加，看起来有一个最大速度，如图2
  • 落地后根据是否连跳、是否移动分别呈现出连跳、移动着地、静止着地的动画，如图3

Celeste

Celeste的跳跃也与按键持续时间正相关，跳跃轨迹是一个正常的抛物线形。其中人物挑起和落地也用了相应的动画进行视觉表现。如果玩家在空中变向，人物则会立刻做出反应并改变速度。此外，Celeste的最大跳跃高度和滞空时间并没有很高的值，主要适用于在整体难度较高的关卡中在各个机关间移动。

图4 Celeste 连续跳跃

上升段：

  • 可以通过按键时间控制跳跃高度
  • 起跳有相应的动画演出
  • 减速过程很快就开始，人物在蹦起一个高度后速度会迅速到0

滞空期：

  • 滞空时间较短，但可以不断连续跳跃

下降段：

  • 明显的下落加速，但速度不像是自由落体

超级马里奥

超级马里奥的跳跃非常经典。一些直观的体验如：“按的久、跳得高”是当时任天堂平台跳跃游戏的标准方案。这个有一定年代感的“老前辈”又与上述的比较新的平台跳跃不同，马里奥的起跳高度控制是非常明显的，例如在起跳长按的过程中甚至能感到人物在空中仍然在加速，而这个加速过程持续一段时间后便会停止，此时玩家在纵向维度上已经无法再控制角色的高度，角色会按惯性继续上升直到最高点。此外，横向存在助跑情况下的大跳滞空时间明显更长，滞空过程中，如果按横向的方向键，角色的速度会有微弱改变，并不会像地面变向那么灵活。

图5 超级马里奥 跳跃

上升段：

  • 可以通过按键时间控制跳跃高度
  • 起跳时保持长按可以感受到加速过程
  • 起跳长按时玩家的初速度并没有完全直接因重力而开始减速，考虑是此时仍然有一个向上的力，重力没有完全发挥作用
  • 如果玩家在上升期间碰到竖直方向的障碍物，会贴边继续上升

滞空期：

  • 滞空时间跟起跳时按键的时间相关
  • 如果起跳有助跑，滞空时间会增加
  • 滞空期间横向的操作有一定的灵活性惩罚，可实现微调但不是完全响应

下降段：

  • 有明显的下落加速
  • 下落疑似有最大速度

分析结论

可以看出，这些具有代表性的平台跳跃游戏在跳跃移动风格不一，但都有一些共同性，结合工程需求，初步设想分“上升”、“下降”两个阶段进行处理，“滞空”则作为上升、下降过程中实际体验的表现，对此作出如下总结。

上升段：

  • 可控的跳跃高度，通过按键时长来实现
  • 短按的小跳应确保有一个最低的跳跃高度，是一个脉冲力的零状态响应
  • 长按的大跳首先叠加小跳的表现，另外会在起跳时会以一个持续力进行加速，加速一段时间后加速失效，开始被重力影响
  • 玩家贴墙跳时能够贴墙上升
  • 记录上升状态、起跳脉冲状态

下降段：

  • 玩家的跳跃行为（纵向分量）在玩家松开跳跃键或上升加速失效后开始受向下的力影响
  • 下落速度有上限，不会超过最大下落速度
  • 记录下落状态、最大速度阈值

滞空期：

  • 可以在空中进行水平方向的控制，如果是转向，则会受Input.GetAxis("Horizontal")带来的惯性阻滞感
  • 根据Demo制作目标，玩家可以在空中衔接攻击动作，但有精确度惩罚

工程实现

全局重力

在实际进行复现之前，不妨先对比Unity刚体的自带重力系统能够实现怎样的跳跃。

图6 自带重力原地跳跃
图7 自带重力跳跃+移动

可见，跳跃的迟滞感非常强，且起跳缺乏速度感，一旦撤去上升力，相当拖沓的滞空时间内玩家完全丧失了纵向维度的控制能力。以及当角色处于高点低速区时，玩家此时能获得更清晰的运动信息，并以此作为落点的判据，而此后由于下落加速过程过于拖沓，伴随着角色水平维度的移动，导致玩家并不能很好地把握落点。如果单纯的增加Gravity Scale来进行加快速度变化，那么由于自带重力系统是全局的，上升和下落基本对称，滞空时间虽然可以缩短，但上升期减速到0的时间也会被压缩，角色被重力束缚的感觉会十分强烈。

改进

那么改进的几个关键方向基本可以确定，首先是起跳要突出向上冲的速度感，再者要能控制跳跃高度，其次滞空时间需要进行控制，最后下落速度要控制在一个最大值之下。

根据分析结论的总结，主要将跳跃分解为跳跃状态、上升参数、下降参数、联动参数几个部分进行设计和复现

跳跃状态

记录玩家是否离地，是否进行起跳。

变量/方法名 类型 描述
IsOnGround() 方法，private bool 利用射线实现检测玩家是否腾空
isJumpUpward 变量，private bool 玩家是否处于上升段
canJumpPulse 变量，private bool 玩家单次起跳中是否可应用脉冲力，一次跳跃只触发一次
isJumpDownward 变量，private bool 玩家是否处于下降段
jumpStartTime 变量，private float 记录玩家起跳离地的时间

上升参数

上升过程分为两个力来控制，一是一次跳跃开始时只触发一次的脉冲力，用于提供上冲的初速度；二是按住跳跃键时持续提供向上的推力，这个推力由一个最大加速时间控制其有效作用的时间范围。

变量/方法名 类型 描述
jumpForce 变量，public float 跳跃上升加速阶段的持续力ForceMode2D.Force
jumpImpulse 变量，public float 跳跃启动时的脉冲力ForceMode2D.Impulse
accelerateTimeLimit 变量，public float 跳跃上升阶段按住跳跃键的最大加速时间

下降参数

联动参数

这部分主要包括如attackJumpPenalty等与战斗、敌人行为相联动的参数，玩家在空中攻击时会受到精准度惩罚，等等这类用于描述跳跃过程中进行攻击等行为的影响因素。

总结

  1. 最好不要用自带重力系统处理平台跳跃，非常不自然
  2. 可以将跳跃分成上升和下降两部分进行分别处理，上升主要注意起跳瞬间的速度、按键时长如何控制高度，下降则主要注意重力模拟、最大下落速度
  3. 滞空表现要和游戏特性对应，跑酷游戏可以稍微长时间、大惯性滞空，战斗游戏则需要短时间、小惯性滞空
  4. 最大加速时间这一做法可以考虑运用上升变加速度到零的处理方法，应该会更自然

除此之外，本次demo还使用了大量碰撞检测、协程、粒子系统，并且各个功能之间实现了初步的联动，最终实现了一个可玩的二维平台跳跃射击小游戏。
